InvestHub Platform Documentation

Comprehensive technical reference manual for the InvestHub Investment Simulation Platform, Version 2.0.

Confidentiality Notice: This documentation details the internal logic, security protocols, and architectural decisions of the InvestHub system. Intended for development and administrative teams only.

1. Executive Summary

InvestHub is a sophisticated simulated investment platform designed to provide a realistic closed-loop financial ecosystem. Users can deposit funds via integrated payment gateways (Razorpay), manage portfolios, and accrue compounded profits based on system-generated simulations.

The system emphasizes integrity, atomicity in financial transactions, and real-time responsiveness via socket-based event propagation. It is built to simulate high-frequency interaction while maintaining a rigid audit trail for all monetary movements.

2. System Architecture

The platform follows a monolithic architecture with modularized service layers, ensuring tight coupling of financial transactions while maintaining separation of concerns for external integrations.

2.1 Core Pillars

  • Security: All routes protected via JWT session tokens; Admin routes enforced via RBAC (Role-Based Access Control).
  • Reliability: MongoDB ACID transactions used for all wallet creation and fund transfers.
  • Scalability: Stateless API design allowing horizontal scaling (excepting the Socket instance which requires Redis adapter for clustering).

4. Authentication & User Management

The system distinguishes strictly between USER and ADMIN roles. Identification is handled via email/password combination with bcrypt hashing.

5. Financial Engine Logic

The core of InvestHub is the handling of user funds. This section details the rigorous logic applied to deposits and withdrawals.

5.1 Deposit Workflow (Razorpay Integration)

  1. Initiation: User requests deposit -> System calls Razorpay Order API -> Returns order_id.
  2. Execution: User completes payment on client side.
  3. Verification: Razorpay webhook/callback hits backend -> Signature verified (HMAC SHA256).
  4. Completion: Wallet balance credited transaction log created.

5.2 Withdrawal & Locking Mechanism

To prevent double-spending, funds requested for withdrawal are immediately moved from Available Balance to Locked Balance.


// Conceptual Logic for Withdrawal Request
async function requestWithdrawal(userId, amount) {
    const session = await mongoose.startSession();
    session.startTransaction();
    
    try {
        const wallet = await Wallet.findOne({ userId }).session(session);
        if (wallet.balance < amount) throw new Error("Insufficient Funds");

        // ATOMIC OPERATION
        wallet.balance -= amount;
        wallet.lockedFunds += amount;
        
        await wallet.save();
        await WithdrawalRequest.create([{ userId, amount, status: 'PENDING' }], { session });
        
        await session.commitTransaction();
    } catch (error) {
        await session.abortTransaction();
        throw error;
    }
}
